## AI 小专栏 #2：退出码与 CI/CD 自动化

### 搜索词: "exit codes CI/CD automation bash script error handling 2026"

**数据点与事实**：
- Bash 中每个命令返回退出码：0 = 成功，非零 = 失败 ([DevOps Roadmap: Bash Scripting](https://medium.com/@sainath.814/devops-roadmap-part-18-bash-scripting-shell-automation-variables-loops-functions-235b85a2f00e))
- 常见退出码：126（无权限）、127（命令未找到）、130（中断信号）([Error handling in bash](https://notifox.com/blog/bash-error-handling))
- CI/CD 最佳实践：使用 `set -e` 让脚本在第一个失败时退出 ([List of exit status codes in Linux](https://medium.com/@himanshurahangdale153/list-of-exit-status-codes-in-linux-f4c00c46c9e0))
- Jenkins 和 Linux 使用退出码来简化流程并增强 CI/CD 管道可靠性 ([Linux and Jenkins: Understanding Exit Codes](https://www.linkedin.com/pulse/linux-jenkins-understanding-exit-codes-smarter-prem-raj-vulli-6a1lc))
- 从 Bash 到 GitHub Actions 的自动化演进：退出码是错误处理的核心 ([From Bash to GitHub Actions: Automating CI/CD](https://infosecwriteups.com/from-bash-to-github-actions-automating-cicd-for-a-real-world-saas-project-d89b251cd371))

**关键洞察**：
- 退出码是脚本和 CI/CD 自动化的"协议"——没有正确的退出码，自动化链路无法判断成功/失败
- 测试工具（如 Playwright）也依赖退出码来报告测试结果 ([Define different exit codes for CI/CD - Playwright issue](https://github.com/microsoft/playwright/issues/14109))
